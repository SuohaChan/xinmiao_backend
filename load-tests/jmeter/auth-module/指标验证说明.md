# 认证模块指标：怎么测、JMeter 能测什么

下面三项常出现在简历/验收里，需区分 **HTTP 压测（JMeter）** 与 **服务端/Redis 观测**。

| 指标 | 含义 | JMeter 能否直接得出「≤5ms / 100%」 | 推荐做法 |
|------|------|-------------------------------------|----------|
| **令牌签发 / 校验耗时 ≤5ms** | 仅指 `JwtUtils` 里 `generateAccessToken` / `verifyAndGetClaims` 的 CPU 级耗时 | **否**。`POST /student/login` 含 **BCrypt**；`GET /student/info` 含 **JWT + 拦截器 + 业务 + DB**；`POST /student/refresh` 含 **多次 Redis**。响应时间 = 网络 + 全链路，**不能**等同「纯 JWT 5ms」。 | **服务端**：在 `JwtUtils` 或 `JwtAuthInterceptor` 对验签/签发包一层 **Micrometer `Timer`**（或用 JFR、Arthas `trace`），看 P99；或单测里对同一密钥做万次签发/校验取耗时。 |
| **权限拦截准确率 100%** | 该放行返回 2xx、该拒绝返回 401/403，且与角色一致 | **能**。用固定账号跑 **正例 + 越权负例**，对 **HTTP 状态码 + 业务 `code`** 断言，汇总 **Error % = 0**。 | 使用本目录 **`auth-metrics-validation.jmx`**（单线程顺序执行，避免竞态）。 |
| **Redis 存储 Refresh Token「命中率」100%** | 通常指 refresh 时 `HGETALL`/`GET` **命中**（或业务上「refresh 全部成功」） | **JMeter 不能直接读 Redis**。脚本只能断言 **`POST /student/refresh` 在 Cookie 合法时 HTTP 200**，这是 **接口成功率**，不是 Redis `keyspace_hits` 比率。 | **Redis**：压测前后执行 `INFO stats`，看 `keyspace_hits` / `keyspace_misses`（需理解你实例上是否只有本业务）；或 **应用埋点**：refresh 路径上 `meterRegistry.counter("auth.refresh.redis.hit")`。 |

## 1. 权限与 Refresh：JMeter 脚本

- **`auth-metrics-validation.jmx`**（本目录）  
  - 学生登录 →（同 Cookie）`POST /student/refresh` → 辅导员登录 →  
    学生访问 `/student/info`（200）、辅导员访问 `/counselor/my/information`（200）、  
    学生访问辅导员接口（403）、辅导员访问学生接口（403）、无 Token / 非法 Token（401）。  
  - **通过标准**：CLI/HTML 报告中 **Error % = 0**（每个取样器上的断言均通过）。

**命令示例**（辅导员账号需存在；与 `LoginDto` 一致）：

```bash
jmeter -n -t load-tests/jmeter/auth-module/auth-metrics-validation.jmx ^
  -JHOST=127.0.0.1 -JPORT=9090 ^
  -Jusername=student1 -Jpassword=123456 ^
  -JcounselorUsername=辅导员账号 -JcounselorPassword=辅导员密码 ^
  -l load-tests/jmeter/results/auth-metrics.jtl -e -o load-tests/jmeter/report/auth-metrics
```

## 2. 令牌 ≤5ms：建议验收表述

- 简历/验收可写：**「Access Token 签发与校验在服务端 Micrometer Timer 中 P99 ≤ 5ms（空载/同机）」**，并附 Grafana/日志截图。  
- **不要**用 JMeter 对 `/student/info` 的 Average/P99 当作「JWT 5ms」，除非已新增**仅验签、不写库**的内部探针接口（需改代码，本仓库默认不提供）。

### 2.1 服务端已实现：`JwtUtils` 上的 Micrometer `Timer`

`xinmiao-server` 里 `JwtUtils` 已注册同名指标 **`jwt.token`**，用标签 **`operation`** 区分（导出到 Prometheus 时点名一般为 `jwt_token_seconds_*`）：

| `operation` | 含义 |
|---------------|------|
| `generate` | `generateAccessToken`：HS256 签名 + `compact()`（**不含** BCrypt） |
| `verify_crypto` | `verifyAndGetClaims` 内 **JJWT 验签 + 读 Claims**（**不含** Redis） |
| `verify_blacklist` | 同一方法里对 **Redis 黑名单** 的 `hasKey`（与「纯验签」分开看尾延迟时有用） |

**看 P99（Prometheus + Grafana）**

1. 启动应用，确保 `application.yml` 里已暴露 `prometheus`（本仓库 `management.endpoints.web.exposure.include` 已含 `prometheus`）。  
2. 拉一点真实流量（登录、带 Token 访问接口），否则分桶为空。  
3. 浏览器或 curl：`http://127.0.0.1:9090/actuator/prometheus`，搜索 `jwt_token_seconds`。  
4. **P99（严谨）**：应用内需对 `jwt.token` 的 **Timer 开启直方图**（本仓库 `JwtUtils` 已 `publishPercentileHistogram()` + `serviceLevelObjectives`，导出后才有 **`jwt_token_seconds_bucket`**）。重启应用并打流量后，在 Prometheus 的 **Graph** 中执行：

```promql
histogram_quantile(0.99, sum by (le, operation) (rate(jwt_token_seconds_bucket[5m])))
```

若 `job` 为 `xinmiao`，可写：

```promql
histogram_quantile(0.99, sum by (le, operation) (rate(jwt_token_seconds_bucket{job="xinmiao"}[5m])))
```

结果单位为 **秒**；若 P99 ≤ `0.005` 即 **≤5ms**。样本量过少时曲线会抖，压测时间略拉长。

5. **平均（不严谨但可写）**：无 `_bucket` 时仍可用 `jwt_token_seconds_sum / jwt_token_seconds_count` 估算平均 ms。

**JFR / Arthas（可选）**

- **JFR**：`JFR.start` 录 `CPU`/`Method Profiling`，看 `JwtUtils` 热点。  
- **Arthas**：`trace com.tree.util.JwtUtils verifyAndGetClaims`（会包含黑名单 Redis，与 `verify_crypto` Timer 区分使用）。

## 3. Redis Refresh「命中率」：建议验收表述

- 若指 **业务**：refresh 请求在合法 Cookie 下 **失败率为 0** → 用 `auth-metrics-validation.jmx` 中 refresh 取样器 + 压测中 `Error %`。  
- 若指 **Redis 键级命中**：在 **Redis** 或 **应用指标** 上取值，与 JMeter 并列写在同一份压测报告里，避免混用术语。
