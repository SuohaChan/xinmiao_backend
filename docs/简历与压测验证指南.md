# 简历技术点 × 压测验证指南（JMeter + 对照实验）

本文档说明：**为支撑简历中的每条技术描述，需要准备什么、测什么、如何用 JMeter 做出可截图对比的数据**。默认本地 **dev** 端口 **`9090`**（见 `application-dev.yml`），路径无全局 `context-path`，完整 URL 形如：`http://127.0.0.1:9090/student/info`。

---

## 一、开始前总清单

### 1.1 环境

| 项 | 要求 |
|----|------|
| JDK / Maven | 与项目一致，可本地启动 `xinmiao-server` |
| MySQL | 库表就绪，有真实数据（资讯 id、通知 id、任务、流水等） |
| Redis | 可连接；需会 **按 key 删除** 或准备 **冷/热两套 id** 做缓存对比 |
| RabbitMQ | 压测若触发任务发布/消费相关链路则需启动 |
| Chroma（向量） | 仅当压测 **带 RAG 的对话** 时需要 |

**通过标准**：同一配置可连续运行 **30 分钟以上** 无随机宕机。

### 1.2 工具

| 工具 | 用途 |
|------|------|
| **JMeter** | 主压测 + 生成 **HTML Dashboard**（曲线、百分位、吞吐量） |
| **hey**（可选） | 快速试并发与 URL，再固定到 JMeter |
| **Postman / curl** | 登录取 Token、验证接口 |
| **redis-cli** | 删除缓存 key、观察 key |

### 1.3 账号与鉴权

- 准备 **学生账号**，能调用 `POST /student/login` 拿到 **Access Token**（响应体）与 **Refresh Token**（Cookie，由 `RefreshTokenCookieHelper` 写入）。
- **JMeter 业务接口**：在 **HTTP Header Manager** 增加：`Authorization: Bearer <access_token>`。
- **Token 过期**：长跑压测前确认 JWT 有效期；过期会导致错误率飙升（非性能问题）。
- **刷新接口** `POST /student/refresh`：依赖 **Cookie 中的 refresh**，JMeter 需配置 **HTTP Cookie Manager**，且先跑登录 sampler 再跑 refresh。

### 1.4 公平对照原则（写简历必守）

1. **两轮对比只改一个变量**（例如：仅「删缓存 vs 不删」或「优化前分支 vs 优化后分支」）。
2. **Thread Group** 的 **线程数、Ramp-up、循环次数** 两轮保持一致。
3. **正式计分前预热**（手动或脚本先请求若干次），避免 JIT/连接池首次尖刺。
4. 记录：**日期、机器配置、线程数、总请求数、Throughput、平均/P95、错误率、本轮条件**。

## 二、JMeter：从 0 到 HTML 报告

### 2.1 安装

1. 安装 JDK 17+。
2. 下载 [Apache JMeter](https://jmeter.apache.org/download_jmeter.cgi)，解压。
3. Windows 运行 `bin/jmeter.bat` 打开 GUI。

### 2.2 最小测试计划结构

```
Test Plan
├── User Defined Variables（可选：BASE_URL、PORT、TOKEN）
├── Thread Group
│   ├── HTTP Request Defaults
│   ├── HTTP Header Manager（Authorization: Bearer ${token}）
│   ├── HTTP Cookie Manager（若需 refresh / 登录链路）
│   ├── HTTP Request（Sampler 1：登录，可选）
│   ├── HTTP Request（Sampler 2：业务接口）
│   └── Listener：Summary Report / Aggregate Report（调试；正式大压测用 CLI 生成 HTML）
```

### 2.3 必配项说明

| 配置项 | 位置 | 建议值（示例） |
|--------|------|----------------|
| **Server Name or IP** | HTTP Request Defaults | `127.0.0.1` |
| **Port** | HTTP Request Defaults | `9090` |
| **Protocol** | HTTP Request Defaults | `http` |
| **Number of Threads** | Thread Group | `20`～`100`（按机器逐步加） |
| **Ramp-up period (seconds)** | Thread Group | `10`～`30`（平滑拉起） |
| **Loop Count** | Thread Group | `50`～`∞`（或固定总请求用 **Throughput Shaping Timer** 插件，进阶） |
| **Authorization** | HTTP Header Manager | `Bearer <access_token>` |
| **Connect / Response Timeout** | 各 HTTP Request | 普通 GET：**3000～10000 ms**；**SSE 流式**：**60000～120000 ms** |

### 2.4 命令行生成「可对比」的 HTML 报告（推荐）

GUI 调通后 **Save** 为 `plan.jmx`，**关闭 GUI** 或大并发时用 CLI：

```bash
jmeter -n -t plan.jmx -l result.jtl -e -o report/run-001
```

- **`-n`**：非 GUI  
- **`-t`**：测试计划  
- **`-l`**：原始结果文件（可删后重跑避免追加混乱）  
- **`-e -o`**：生成 **`report/run-001/index.html`**

**对比流程**：

1. 条件 A：生成 `report/baseline`  
2. 只改变一个条件（或换分支部署）  
3. 再生成 `report/optimized`  
4. 浏览器打开两个 **`index.html`**，对照 **Throughput、Response Times Percentiles、Errors %**，截图保存。

### 2.5 报告中重点看图

| 图表/指标 | 含义 |
|-----------|------|
| **Throughput** | 吞吐（约等于 QPS 量级） |
| **90% / 95% / 99% Line** | 尾延迟，适合写「优化后 P95 下降」 |
| **Average** | 平均响应时间 |
| **Errors %** | 鉴权失败、限流、超时、503 等 |

### 2.6 注意事项

- **View Results Tree** 仅调试时开，大压测关闭，否则占内存。  
- **SSE**：JMeter 会等响应结束才算完成，耗时长；**并发宜低**，超时设大。  
- **WebSocket**：标准 HTTP Sampler **不能**替代长连接；需 **WebSocket 插件** 或 Postman/脚本单独测（见下文）。

## 三、按简历条目：要验证什么、怎么测、JMeter 怎么配

以下接口路径以当前工程 **student** 包为准（无前缀 `/api` 时即为根路径）。

---

### 3.1 认证授权（JWT + Refresh + Redis + 拦截器 + ThreadLocal + 角色）

**简历要支撑的说法**：无状态 Access Token；Refresh 存 Redis；续期与登出/黑名单；拦截器解析 JWT；ThreadLocal 传用户；按角色控制接口。

| 验证点 | 怎么测 | JMeter / 工具 |
|--------|--------|----------------|
| **带 Token 访问成功** | `GET /student/info` 带 `Authorization: Bearer` | Thread Group + Header Manager；看 **错误率 0** |
| **无 Token / 错 Token** | 去掉 Header 或填 `Bearer invalid` | 单独 HTTP Request；期望 **401**，错误率 **100%** 或断言状态码 |
| **Refresh** | `POST /student/refresh`，需 Cookie | **HTTP Cookie Manager** + 先 **POST /student/login**（JSON Body）提取 Cookie；再 refresh；看 **200** 与响应体新 token |
| **强制下线** | 登出后原 Access 仍短时可访问、Refresh 失效等 | **手动**：logout 后再 refresh，期望失败；或查 Redis 中 refresh key 是否删除（文档化截图） |
| **QPS「对比」** | 更适合对比 **优化前后** 或 **鉴权开销** | 同一 `GET /student/info`，两轮相同线程配置，对比 **Throughput / P95**（若仅鉴权无代码变更，差异可能不大，简历侧重「机制」即可） |

**JMeter 参数小结**：业务读接口 **Timeout 5～10s**；登录 **POST** `Content-Type: application/json`。

---

### 3.2 缓存与高并发（Cache-Aside、空值、分布式锁、TTL 抖动、回源限流、对话 Redis JSON）

**简历要支撑的说法**：穿透/击穿/雪崩治理；Redis 异常回源限流；会话记忆 JSON + 条数截断。

| 验证点 | 怎么测 | JMeter / 操作 |
|--------|--------|----------------|
| **热缓存 vs 冷缓存** | 固定资讯 id：`GET /student/informations/{id}` 或通知 `GET /student/notices/{id}` | **Round A**：`redis-cli DEL` 对应 `xinmiao:cache:information:<id>`（或通知 cache key）后压测 → **Round B**：不删直接压。对比 **Throughput↑、Average↓** |
| **空值缓存** | 使用**不存在但合法格式**的 id（若返回 404 且写了空值缓存） | 观察第二次同 id 是否仍打库（可结合日志/SQL 日志，简历写「验证思路」） |
| **回源限流** | 高并发 + 大量 miss | 观察 **503 / SERVICE_BUSY** 是否出现、`Errors %`；与 `app.cache.fallback` 配置一致 |
| **对话记忆** | 多轮对话后 Redis `chat:memory:*` | **功能验证为主**；QPS 对比优先用普通 GET，对话接口瓶颈常在 LLM |

**对比写法（简历）**：「在相同 JMeter 线程配置下，资讯详情缓存命中后吞吐提升至约 **×** 倍，P95 延迟下降约 **×%**（测试环境：单机 dev）。」

---

### 3.3 AI 能力（Spring AI、SSE、Chroma RAG、线程池隔离、Semaphore、超时、Redis 滑动窗口限流）

**简历要支撑的说法**：流式输出、RAG、线程池隔离、并发与超时、按用户限流。

| 验证点 | 怎么测 | JMeter / 说明 |
|--------|--------|----------------|
| **SSE 流式** | `POST /student/chat/stream`，`Content-Type: application/json`，Body：`{"message":"你好"}` | **单请求 Timeout 60～120s**；**并发低**（受 `app.chat.stream.max-concurrent` Semaphore 限制）；看 **响应时间分布**（整段流结束才结束） |
| **非流式** | `POST /student/chat` | 同样带 Token；超时略短于 SSE |
| **限流** | 同一用户高并发打 chat | **提高线程数**，观察 **429/业务错误** 与 `Errors %`；报告体现「限流生效」 |
| **RAG** | Header `Use-RAG: true` | 需 Chroma 可用；延迟会大于无 RAG，**不要**与纯 GET 比绝对 QPS |

**简历表述建议**：AI 段写 **「高并发下稳定性（限流 + 并发上限 + 超时）」**，少写「QPS 翻倍」，避免与 LLM 瓶颈矛盾。

---

### 3.4 积分与排行榜（Set 防重、CAS、ZSet、Lua 增量、定时重建）

**简历要支撑的说法**：任务完成幂等、多维度榜、Lua 更新、最终一致。

| 验证点 | 怎么测 | JMeter |
|--------|--------|--------|
| **读榜 QPS** | `GET /student/tasks/rankings/today`、`.../week`、`.../college` | 标准 GET + Token；**热榜**多轮压测对比 **Throughput** |
| **提交任务幂等** | `POST /student/tasks/complete`（Body 按 DTO） | **重复相同 taskId** 两轮，结合 DB/业务结果验证幂等（功能为主）；压测看 **错误率与延迟** |
| **最终一致** | 定时任务重建 | **线下**：改数据后等任务跑完对比榜单；**压测报告**侧写「读路径性能」即可 |

---

### 3.5 实时消息（WebSocket + STOMP、JWT、RabbitMQ、重试、DLQ、心跳）

| 验证点 | 怎么测 |
|--------|--------|
| **HTTP 压测** | 仅覆盖「触发推送的 **业务接口**」；**WS 连接本身**需 **JMeter WebSocket 插件** 或 **浏览器/脚本** |
| **MQ** | JMeter 压「发布任务」类接口，**RabbitMQ 管理台**看队列积压、DLQ |
| **简历** | 写清「接口压测 + 管理台截图 + 握手鉴权为手工/自动化验证」 |

---

### 3.6 工程化（全局异常、Validation、分组校验）

| 验证点 | 怎么测 |
|--------|--------|
| **错误码统一** | JMeter 对非法 Body **断言** HTTP 400 与统一 JSON 结构 |
| **分组校验** | 对不同场景传缺字段，看返回与日志 |

## 六、简历中如何写「JMeter 对比」一句话模板

- 「在 **固定并发（如 50 线程）与相同总请求数**下，对 `GET ...` 分别于 **缓存未预热** 与 **缓存命中** 各执行一轮 JMeter 压测，**吞吐提升至约 × 倍，P95 延迟从 A ms 降至 B ms**（dev 单机、MySQL+Redis）。」  
- 「对对话接口提高并发，JMeter 报告显示 **错误率随并发上升**，与 **Redis 滑动窗口限流** 行为一致。」
